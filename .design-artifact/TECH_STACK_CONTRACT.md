# Tech Stack & API Contract — German Master Application Tracker

## 1. Architecture

```
Browser (React, rendered by Next.js)
   │  HTTP / SSE (cookie session)
   ▼
Next.js app (frontend + thin backend-for-frontend)
   │  REST, internal service-to-service auth (JWT/API key)
   ├──────────────► DB-service (Node/Fastify or NestJS)
   │                     │
   │                     ▼
   │                 Supabase (Postgres + Auth + Storage)
   │
   └──────────────► AI-service (Node agent)
                         │  REST (service token)
                         ▼
                    DB-service (read context, optionally write via tool-calls)
                         │
                         ▼
                    Claude API
```

- **Frontend never calls DB-service or AI-service directly.** All browser traffic goes through Next.js route handlers / Server Actions, which attach the authenticated user's session and role before forwarding.
- **Next.js ↔ DB-service** and **Next.js ↔ AI-service**: plain REST/JSON over HTTPS, versioned (`/api/v1/...`), authenticated with a server-to-server secret (not the end-user's session).
- **AI-service ↔ DB-service**: same pattern, using a distinct `service:ai-agent` credential so writes made by the agent are attributable and auditable separately from user-initiated writes.
- **Supabase** is the source of truth: Postgres (data), Supabase Auth (Applicant email magic-link), Supabase Storage (library files + "Other files").

## 2. Services & Responsibilities

| Service | Tech | Responsibility |
|---|---|---|
| **Frontend** | Next.js (App Router) + React | UI, session cookie handling, calls own API routes only |
| **DB-service** | Node + Fastify (or NestJS) | CRUD for all domain data, role enforcement, talks to Supabase via service-role key (never exposed to frontend) |
| **AI-service** | Node + Claude SDK | Builds grounded system prompt from context, calls Claude, streams response, optionally executes agent tool-calls back into DB-service |
| **Supabase** | Postgres, Auth, Storage | Persistence, applicant auth, file storage |

## 3. Auth Model

- **Applicant**: Supabase Auth, email magic-link. Full read/write.
- **Family / Counselor**: no Supabase Auth user — they authenticate via an **access token** issued by the Applicant.
  - Token is generated by DB-service, stored **hashed** (`sha256`) in `access_tokens`, never stored in plaintext server-side.
  - On redemption, DB-service issues a signed session cookie (JWT, short-lived + refresh) scoped to `{ role: 'family'|'counselor', applicant_id }`.
  - Regenerating a token invalidates the old hash immediately; existing cookie sessions remain valid until they expire or the user logs out (matches the prototype's stated behavior).
- **Service-to-service**: static bearer secrets in environment variables (`DB_SERVICE_TOKEN`, `AI_SERVICE_TOKEN`), rotated manually or via secret manager. Not user-derived.

## 4. Database Schema (Supabase / Postgres)

```sql
-- Applicants (1 per Supabase auth user)
create table applicants (
  id uuid primary key default gen_random_uuid(),
  auth_user_id uuid references auth.users(id) unique not null,
  email text not null,
  created_at timestamptz default now()
);

create table access_tokens (
  id uuid primary key default gen_random_uuid(),
  applicant_id uuid references applicants(id) on delete cascade not null,
  role text check (role in ('family','counselor')) not null,
  token_hash text not null,
  created_at timestamptz default now(),
  revoked_at timestamptz,
  unique (applicant_id, role)
);

create table universities (
  id uuid primary key default gen_random_uuid(),
  applicant_id uuid references applicants(id) on delete cascade not null,
  name text not null,
  program text,
  city text,
  gpa_req text,
  language_req text,
  aps_required boolean default true,
  tuition_per_semester numeric default 0,
  deadline date,
  status text check (status in ('Not started','Researching','In progress','Submitted','Offer')) default 'Not started',
  link text,
  created_at timestamptz default now()
);

create table documents (
  id uuid primary key default gen_random_uuid(),
  applicant_id uuid references applicants(id) on delete cascade not null,
  name text not null,
  category text not null,
  done boolean default false
);

create table library_files (
  id uuid primary key default gen_random_uuid(),
  applicant_id uuid references applicants(id) on delete cascade not null,
  slot_key text,               -- null for free-form "Other files"; set for fixed slots (e.g. 'bachelor_cert')
  category text not null,      -- 'german' | 'prep' | 'certs' | 'other'
  name text not null,
  storage_path text,           -- Supabase Storage object path
  size_bytes bigint,
  uploaded_at timestamptz default now()
);

create table costs (
  applicant_id uuid primary key references applicants(id) on delete cascade,
  blocked_account_year numeric default 11904,
  semester_fee numeric default 250,
  aps_fee numeric default 200,
  uni_assist_fee numeric default 75,
  visa_fee numeric default 75,
  tuition_per_semester numeric default 0,
  health_insurance_monthly numeric default 110,
  months int default 12
);

create table scholarships (
  id uuid primary key default gen_random_uuid(),
  applicant_id uuid references applicants(id) on delete cascade not null,
  name text not null,
  provider text,
  amount text,
  deadline date,
  link text,
  status text check (status in ('Not applied','In progress','Applied','Awarded')) default 'Not applied'
);

create table timeline_events (
  id uuid primary key default gen_random_uuid(),
  applicant_id uuid references applicants(id) on delete cascade not null,
  label text not null,
  date date not null,
  source text check (source in ('university','scholarship','custom')) not null,
  source_id uuid  -- null for custom
);

create table chat_messages (
  id uuid primary key default gen_random_uuid(),
  applicant_id uuid references applicants(id) on delete cascade not null,
  role text check (role in ('user','assistant')) not null,
  content text not null,
  created_at timestamptz default now()
);
```

### Row Level Security (enforced at DB-service layer, mirrored in Supabase RLS as defense-in-depth)

```sql
alter table universities enable row level security;
create policy "applicant owns data" on universities
  for all using (applicant_id = current_setting('app.current_applicant_id')::uuid);
-- repeat per table; family/counselor sessions carry the same applicant_id claim but DB-service
-- enforces read-only at the API layer (RLS alone doesn't know about "read-only role" — that's an app-level check).
```

> RLS here scopes rows to the correct applicant; **role-based read-only enforcement (family/counselor can't write) is done in DB-service**, not in Postgres — Supabase RLS isn't the place to encode "this role can SELECT but not UPDATE" per request without custom claims plumbing, and DB-service already validates role on every mutating route.

## 5. API Contract

Base path: `/api/v1`. All requests authenticated via session cookie (browser→Next.js) or bearer service token (service→service). Responses `application/json`. Errors: `{ error: { code, message } }`.

### Next.js → DB-service

| Method | Path | Roles | Body / Notes |
|---|---|---|---|
| GET | `/api/v1/universities` | all | — |
| POST | `/api/v1/universities` | applicant | `{ name, program, city, gpaReq, languageReq, apsRequired, tuitionPerSemester, deadline, link }` |
| PATCH | `/api/v1/universities/:id` | applicant | partial update (status, deadline, etc.) |
| DELETE | `/api/v1/universities/:id` | applicant | |
| GET | `/api/v1/documents` | all | |
| PATCH | `/api/v1/documents/:id` | applicant | `{ done: boolean }` |
| GET | `/api/v1/library` | all | returns fixed-slot docs + other-files list |
| POST | `/api/v1/library/upload` | applicant | multipart; `{ category, slotKey? }` → uploads to Supabase Storage, inserts row |
| DELETE | `/api/v1/library/:id` | applicant | removes storage object + row |
| GET | `/api/v1/costs` | all | |
| PUT | `/api/v1/costs` | applicant | full cost object replace |
| GET | `/api/v1/scholarships` | all | |
| POST | `/api/v1/scholarships` | applicant | |
| PATCH | `/api/v1/scholarships/:id` | applicant | `{ status }` or `{ deadline }` |
| GET | `/api/v1/timeline` | all | server merges university + scholarship + custom, sorted |
| POST | `/api/v1/timeline/custom` | applicant | `{ label, date }` |
| PATCH | `/api/v1/timeline/custom/:id` | applicant | |
| DELETE | `/api/v1/timeline/custom/:id` | applicant | |
| POST | `/api/v1/access-tokens/:role` | applicant | generates/regenerates token, returns plaintext **once** |
| POST | `/api/v1/access-tokens/redeem` | public (rate-limited) | `{ token }` → sets session cookie for family/counselor |
| POST | `/api/v1/auth/logout` | all | clears session cookie |

### Next.js → AI-service

| Method | Path | Notes |
|---|---|---|
| POST | `/api/v1/chat` | `{ message, applicantId, role, lang }` → Next.js route handler streams back SSE to browser; internally calls AI-service which fetches live context from DB-service before calling Claude |

### AI-service → DB-service

| Method | Path | Notes |
|---|---|---|
| GET | `/internal/v1/context/:applicantId` | returns compact snapshot (universities, costs, scholarships) for prompt-building |
| POST | `/internal/v1/*` | only if the agent has tool-calling write access; requires `service:ai-agent` token, logs `actor: 'agent'` on the row |

## 6. Environment Variables (per service)

**Next.js**
```
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
DB_SERVICE_URL=
DB_SERVICE_TOKEN=
AI_SERVICE_URL=
AI_SERVICE_TOKEN=
SESSION_JWT_SECRET=
```

**DB-service**
```
SUPABASE_URL=
SUPABASE_SERVICE_ROLE_KEY=   # server-only, full access, bypasses RLS — guard carefully
DB_SERVICE_TOKEN=            # validates incoming calls from Next.js / AI-service
SESSION_JWT_SECRET=          # same value as Next.js, to verify session cookies if DB-service is hit directly
```

**AI-service**
```
ANTHROPIC_API_KEY=
AI_SERVICE_TOKEN=
DB_SERVICE_URL=
DB_SERVICE_TOKEN=            # reused, or a distinct service:ai-agent credential — recommended: distinct
```

- DB-service → **Render** or **Railway** free web service
- AI-service → **Render** or **Railway** free web service (separate instance from DB-service for independent scaling/rate-limits)
- Supabase → free project tier (500MB DB, 1GB storage, 50k monthly active auth users)

## 8. Open Decisions To Confirm

1. Does the AI-agent need **write** tool-calls (e.g. "add this milestone for me"), or read-only context for now? Changes whether `service:ai-agent` needs write policies at all.
2. Session cookie strategy: single JWT with `role` claim, or Supabase's own session for applicant + a custom cookie for family/counselor? (Recommended: custom lightweight JWT for both, so DB-service has one verification path.)
3. Rate-limiting on `/access-tokens/redeem` (public, brute-forceable) — suggest IP + token-prefix throttling at the edge (Vercel Edge Config or Upstash Redis).
